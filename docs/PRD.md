## 1. 제품 비전 (Product Vision)

> **「글보다는 한 장의 사진으로 말하는, 나만의 미식 지도 서비스」**

사용자가 방문한 식당을 **지도 위 한 점(마커)** 으로 남기고, 그 자리에 **사진**을 쌓아 개인 아카이브를 만든다. 긴 텍스트 리뷰보다 **시각적 기록**과 **짧은 감상**에 집중한다.

---

## 1.1 최우선 제품 원칙 (Camera-First)

| 우선순위 | 내용 |
|----------|------|
| **핵심 경험** | 서비스에서 **가장 중요한 기능**은 “기기 **갤러리**에서 앨범 사진을 고르는 것”이 **아니다.** |
| **정의** | 사용자가 **그 자리에서 카메라로 촬영한 사진만** 간단히 **편집(크롭·보정 등)** 한 뒤 올리는 흐름을 **1순위**로 설계한다. → **즉시 촬영 → (선택) 편집 → 업로드**가 기본 플로우다. |
| **갤러리·파일 업로드 (명시적 비제공)** | 디바이스 **갤러리·앨범에서 사진을 고르는 기능**과, 데스크톱 **드래그 앤 드롭으로 파일을 올리는 기능**은 **도입하지 않는다.** (초기만이 아니라 **제품 로드맵 전체에서 제공하지 않음**.) |

이 원칙은 **진정성 있는 “지금 이 식당” 기록**과 **촬영·편집 UX**에 리소스를 몰아주기 위함이다.

## 2. 문제 정의 & 목표

| 구분 | 설명 |
|------|------|
| **문제** | 기존 맛집 앱은 텍스트·별점 중심이라, “그날의 분위기”와 “내가 본 그 한 끼”를 담기 어렵다. |
| **목표** | 지도 + **방문 사진 아카이브**(앱 내 모아보기) + 가벼운 소셜(공유)로 **퍼스널 미식 다이어리**를 제공한다. (여기서 “갤러리”는 기기 앨범이 아니라 **서비스 안 사진 모음**을 뜻함.) |
| **비목표 (초기)** | 전국 DB 맛집 랭킹, 배달 연동, 결제 등은 범위에서 제외할 수 있다. |
| **비목표 (영구)** | **갤러리·앨범에서의 사진 선택 업로드**, **드래그 앤 드롭 파일 업로드** — [1.1](#11-최우선-제품-원칙-camera-first)과 동일하게 **제공하지 않음**. |

## 2.5 로그인 방식 (Authentication)

| 항목 | 내용 |
|------|------|
| **방식** | **카카오 로그인**(Kakao Login) — 사용자는 **카카오톡 계정**으로 서비스에 로그인한다. |
| **목적** | 마커·사진·코스 등 **개인 데이터**를 계정에 귀속하고, 공유 링크·다기기 접근 시 동일 사용자로 인식한다. |
| **제품 문구** | 온보딩·버튼 등에는 사용자에게 익숙한 **「카카오톡으로 로그인」** / **「카카오로 시작하기」** 류의 카피를 사용한다. (실제 연동은 Kakao Developers의 **카카오 로그인** 제품) |
| **구현 시 유의** | [Kakao Developers](https://developers.kakao.com) 앱에 **카카오 로그인** 활성화, **Redirect URI**, **동의항목**(닉네임·프로필 이미지 등 필요 시) 설정. 백엔드 또는 BFF에서 **액세스 토큰 검증** 후 자체 세션·JWT 발급 또는 **Supabase Auth의 Kakao 프로바이더** 등과 조합 가능. |
| **수용 기준 (예시)** | 로그인 성공 후 이전에 저장한 마커·피드가 동일 계정으로 조회된다. 로그아웃 시 민감 작업(업로드·삭제)이 제한된다. |

## 3. 핵심 기능 (Key Features)

### 3.1 맵 기반 방문 인증 (Marker & Map)

| 항목 | 내용 |
|------|------|
| **기능** | 지도에서 **특정 위치 롱탭(길게 누르기)** 또는 **주소 검색**으로 “방문 식당” 마커를 생성한다. |
| **UI/UX** | 애플 지도에 가깝게 **깔끔하고 미니멀한 커스텀 마커** 디자인을 적용한다. |
| **수용 기준 (예시)** | 롱탭 후 마커 생성까지 1회 성공 플로우가 끊기지 않아야 한다. 주소 검색 실패 시 명확한 피드백을 준다. |

### 3.2 퍼스널 맛집 갤러리 (Visual Archive)

| 항목 | 내용 |
|------|------|
| **기능** | 마커를 탭하면 해당 식당에서 남긴 사진이 **카드 형태**로 펼쳐지고, **하단 시트(Bottom Sheet)** 로 표시한다. |
| **업로드 (유일한 경로)** | **[1.1 최우선 제품 원칙](#11-최우선-제품-원칙-camera-first)** 에 따라, **카메라로 방금 찍은 사진**만 촬영 후 **앱 내 편집**(크롭·회전·밝기 등 최소 기능)을 거쳐 업로드하는 **인라인 카메라 플로우**만 제공한다. 저장소는 **Supabase Storage** 연동을 권장한다. |
| **제공하지 않음** | 갤러리·앨범 피커, **드래그 앤 드롭**으로의 사진 추가는 **비목표(영구)** — [2. 문제 정의 & 목표](#2-문제-정의--목표) 참고. |
| **수용 기준 (예시)** | **촬영 → 편집 → 업로드**가 3회 이내 탭으로 끝나도록 설계한다. 시트 내 스크롤·닫기 동작이 모바일·데스크톱 모두 자연스러워야 한다. |

### 3.3 사진 중심의 소통 (Visual Communication)

| 항목 | 내용 |
|------|------|
| **기능** | 긴 리뷰 대신, 사진 위·옆에 **짧은 태그** 또는 **한 줄 평**만 노출한다. |
| **공유** | 사용자가 만든 **마커 그룹**(예: “안산 빵지순례 코스”)을 **링크 하나**로 친구에게 공유한다. |
| **수용 기준 (예시)** | 공유 링크 수신자는 로그인 없이(정책에 따라) 지도·사진 미리보기가 가능해야 한다. |

### 3.4 이미지 그리드 피드 (Photo-First Feed)

| 항목 | 내용 |
|------|------|
| **기능** | 지도 외에, 내가 올린 사진만 모아 **핀터레스트 스타일 그리드 뷰**를 제공한다. |
| **수용 기준 (예시)** | 그리드에서 항목 탭 시 해당 마커/식당 상세(또는 시트)로 이동한다. |

## 4. 디자인 가이드 (Apple-Like Aesthetics)

서비스 전반에 **“애플처럼 깔끔한”** 인상을 주기 위한 지침이다.

| 요소 | 설명 | 구현 힌트 (예) |
|------|------|----------------|
| **Glassmorphism** | 사진 위 정보창·오버레이는 **반투명 유리** 질감(블러 + 얇은 테두리). | `backdrop-blur`, 투명도·그라데이션 조합 |
| **Rounded Corners** | 카드·버튼·시트 모서리를 **충분히 둥글게** (예: `rounded-2xl` 수준). | 터치 영역과 시각적 무게 균형 유지 |
| **Haptic Feedback (느낌)** | 마커 생성·사진 업로드 시 **미세한 애니메이션**(스케일·페이드 등)으로 피드백. | 모바일 네이티브 햅틱은 선택, 웹은 CSS/모션으로 대체 |

## 5. 정보 구조 & 화면 흐름 (요약)

```
[온보딩 / 계정]
  └─ 카카오톡(카카오) 로그인 → 세션 확보

[지도 화면]
  └─ 롱탭 / 주소검색 → 마커 생성
  └─ 마커 탭 → Bottom Sheet (사진 목록 + 태그·한줄평)
  └─ 사진 추가: **카메라 촬영 → (편집) → 업로드** 만 (갤러리·드래그앤드롭 없음)

[피드 화면]
  └─ 내 사진 그리드 → 상세/마커 연결

[공유]
  └─ 코스(마커 그룹) → 공유 URL
```

## 6. 기술적 도전 과제 (학습·구현 포인트)

팀·수업에서 함께 풀 수 있는 주제로 정리한다.

| 주제 | 질문 | 권장 방향 |
|------|------|-----------|
| **사진 저장소** | 사진을 어디에 둘 것인가? | **Supabase Storage** + 공개/비공개 버킷 정책 |
| **데이터 구조** | 식당 정보 · 방문일 · 사진 URL · 사용자 ID를 어떻게 연결할 것인가? | RDB 스키마 설계(예: `users`, `places`, `visits`, `photos`, `collections`). **사용자 ID**는 카카오 로그인 연동 후 발급되는 내부 UUID 등과 매핑 |
| **이미지 최적화** | 고용량 원본이 지도·피드 성능을 해치지 않게 하려면? | **썸네일** 생성(클라이언트 리사이즈 또는 서버/Edge 처리), 지도에는 썸네일·상세에서 원본 |
| **카메라·편집** | 웹/앱에서 “지금 찍기”만 허용하려면? | 웹: `getUserMedia` / `capture="environment"` 등 **카메라 입력** + Canvas·Web API로 **편집 후 Blob 업로드**. 네이티브: 카메라 Intent·권한·촬영 전용 UI. **갤러리 피커·`<input type="file">` 다중 선택·드래그앤드롭 존은 구현하지 않음** (제품 비목표) |

## 7. 의존성 & 제약 (초안)

| 항목 | 메모 |
|------|------|
| 지도 | 카카오맵 등 기존 지도 SDK 정책·쿼터 준수 |
| **로그인** | **카카오 로그인** 필수 정책 시, 동의항목·Redirect URI·보안 저장(Client Secret 등) 준수 |
| 인증·연동 | Supabase Auth + Kakao 프로바이더 또는 자체 OAuth 콜백 처리 시 **카카오 사용자 식별자**와 앱 `users` 테이블 매핑 일관성 유지 |
| 개인정보 | 위치·사진 EXIF·카카오 프로필 동의 범위 검토 |
| 카메라·미디어 | 브라우저·OS **카메라·마이크(필요 시) 권한** 안내, HTTPS 환경 등 [보안 컨텍스트](https://developer.mozilla.org/en-US/docs/Web/Security/Secure_Contexts) 요구 |
| 업로드 UX | **카메라 촬영 경로만** 노출; 갤러리·드래그앤드롭 UI는 설계·개발 범위에 넣지 않음 |

## 8. 성공 지표 (예시 — 추후 수치 확정)

- 마커 생성 후 **카메라 촬영 → 업로드** 완료 전환율 (갤러리 경로는 없음)

- 피드 화면 체류 시간
